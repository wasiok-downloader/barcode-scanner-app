API → UI mapping

Open Food Facts fields, in the order they appear in the report.

1 · Product hero

  • product_name → title
  • image_url → photo (falls back to monogram)
  • nutrition_grades → score badge (A–E; "—" if NA / not-applicable)

2 · Personal alerts

Built by matching API data against the user's profile (allergies, goals, health risks). Each alert is one API field matched with one profile rule.

  • API allergens_tags matches the user's allergy list → danger alert (e.g. "Hazelnuts" for tree-nut allergy)
  • API traces_tags matches the user's allergy list → warn alert ("may contain")
  • A nutrient (sugars_100g / saturated-fat_100g / salt_100g) is above its threshold and the user has a related goal (weight loss, diabetes, heart) → severity by level
  • Product has additives (additives_tags not empty) or nova_group ≥ 4, and the user wants to avoid ultra-processed food → info alert
  • Nothing to match (no allergens, no nutriments) → optional generic info alert ("No nutritional data — refer to label", as in the No-score demo)

In this prototype alerts are hardcoded per product to simulate the matched output. In production, derived per request from user profile + API response.

3 · Breakdown rows

  • nutriments.sugars_100g / saturated-fat_100g / salt_100g → Sugar / Sat fat / Salt rows. API gives the number only; Low/Med/High and the explanation copy are computed in the prototype from static EU/WHO thresholds — needs an owner (PM, nutritionist, or backend).
  • nova_group → Processing row (1–4). API gives the digit; description ("Ultra-processed…") is static copy.
  • additives_tags[] → Additives row (empty array → "None").

4 · Ingredients

  • ingredients_text_en → split by commas; tokens highlighted by alert match (allergens get a coloured background).

Missing-data rule

If a field is absent in the API response, the corresponding row or section is hidden entirely — no "no data" placeholders.
